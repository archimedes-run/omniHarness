# Contributing to Archimedes OmniHarness

Welcome! We are thrilled you want to contribute to OmniHarness. This project is a production-grade, LangGraph-based AI super-agent framework designed for autonomy, sandboxed code execution, and secure orchestration. 

To ensure the codebase remains clean, secure, and highly modular, please review this guide before submitting a Pull Request.

---

## 🏛️ The Golden Rule: Architecture Boundary

The most critical architectural constraint in this repository is the strict separation between the **Harness** and the **App**. 

* **The Harness** (`backend/packages/harness/omniharness/`): This is a standalone, publishable Python library. It contains the core LangGraph orchestrator, `AgentMiddleware` pipeline, Pydantic configuration schemas, and native tool execution logic.
  * 🛑 **CRITICAL:** The harness layer must **NEVER** import anything from the `app.*` layer.
* **The App** (`backend/app/`): This is the implementation layer. It contains the FastAPI gateway (`app/gateway/`), messaging bridges (`app/channels/`), and custom tools/middlewares. 

If you are adding a highly specific integration (e.g., OpenClaw channel, a custom OpenClaude CLI wrapper), it belongs in the `app/` layer. We enforce this boundary in CI via `test_harness_boundary.py`.

## 💻 Local Development Setup

### Prerequisites
* **Python**: `3.12`
* **Node.js**: `22` (Corepack enabled for `pnpm`)
* **Docker**: Required for the `AioSandboxProvider` local container execution.

### Backend Setup

We use `uv` for fast Python package management.

```bash
cd backend
uv sync --group dev
```

### Frontend Setup

We use a pinned version of pnpm (10.x).

```bash
cd frontend
corepack enable
corepack prepare pnpm@10.26.2 --activate
pnpm install --frozen-lockfile
```

## 🛠️ Development Guidelines

### Backend (Python)

**Linting and Formatting:** We use ruff. Run `make lint` in the backend/ directory before committing.

**Type Hinting:** Strict type hinting is enforced. Use Pydantic BaseModel for all data structures and configuration schemas.

**Adding Middlewares:** If you are adding a custom middleware, extend AgentMiddleware and use the @Prev or @Next decorators to inject it into the LangGraph pipeline relative to existing canonical middlewares (e.g., `@Prev(OmniHarnessSummarizationMiddleware)`).

**Adding Tools:** Custom tools should be placed in `backend/app/tools/`. Ensure you define an explicit args_schema using Pydantic. Expose a module-level instance of the tool so it can be loaded dynamically via `config.yaml` using the `resolve_variable()` syntax (e.g., `use: app.tools.my_tool:my_tool_instance`).

### Frontend (Next.js / React)

**Styling Rules:** We strictly use the Tailwind `stone-*` color palette (e.g., `bg-stone-800`, `text-stone-200`). Do not use `slate-*` or `gray-*`.

**No Emojis:** Please refrain from using emojis in UI badges or system alerts. Use uppercase typography and clear text instead (e.g., `[LOOP DETECTED]`).

**Type Safety:** The CI pipeline runs `pnpm typecheck`. Ensure all React props and state are strictly typed. Avoid `any`.

**Formatting:** Run `pnpm format` and `pnpm lint` before pushing.

## 🧪 Testing

We require tests for all new features and bug fixes.

* Run backend tests using pytest in the backend/ directory.
* Ensure that the `test_harness_boundary.py` test passes if you modify cross-module imports.
* If you modify LLM streaming behaviors, ensure you do not raise destructive exceptions within `on_llm_new_token`. Use LangGraph state mutations (like returning `RemoveMessage` in a state update) instead.

---

## 📝 Pull Request Process

**Fork & Branch:** Fork the repository and create a new branch from main (e.g., `feature/openclaw-channel` or `fix/sandbox-timeout`).

**Commit Messages:** Write clear, concise commit messages.

**Update Documentation:** If you add a new configuration parameter or tool, update the README.md or relevant configuration examples (config.example.yaml).

**Pass CI:** Open a Pull Request and ensure GitHub Actions (Lint Check) pass. The CI will verify:

* `uv sync` and `make lint`
* `pnpm format`, `pnpm lint`, and `pnpm typecheck`

**Review:** A maintainer will review your code. Be prepared to address feedback, especially regarding security (sandbox escapes) and architectural boundaries.

---

## 🔐 Security & Sandboxing

Because this framework executes code generated by LLMs, security is paramount.

* Never use `shell=True` without `shlex.quote`.
* Respect Virtual Paths: Any new host-side tool must strictly operate within `/mnt/user-data/...` or `/mnt/acp-workspace/`. Do not allow directory traversal out of the workspace.
* Guardrails: If your feature involves high-risk system commands, consider hooking into the `GuardrailMiddleware` rather than building custom execution blockers.
